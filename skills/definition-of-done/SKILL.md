# Definition of Done

每个任务完成后都需要满足的项目级标准。不同于"验收条件"（因任务而异），完成标准是固定的质量底线。

## Definition of Done vs. Acceptance Criteria

| | Acceptance Criteria | Definition of Done |
|---|---|---|
| **范围** | 针对单个任务或规格 | 应用于每个增量 |
| **变化** | 每个任务不同 | 固定不变 |
| **回答** | "我们建对了吗？" | "这能交付吗？" |
| **所有权** | 在规划时定义 | 项目级标准 |
| **例子** | "用户能重置密码" | "测试通过，无回归，文档已更新" |

**关键**：任务只有同时满足**验收条件**和**完成标准**才算完成。

## 完成标准清单

在宣布任务完成前，确保以下所有项都满足：

### 正确性（Correctness）

- [ ] 所有验收条件都满足了
- [ ] 代码能运行，行为符合预期（非仅编译或类型检查）
- [ ] 新增的功能由测试覆盖：测试在修改前会失败，修改后会通过
- [ ] 现有测试仍然通过（无回归）
- [ ] 边界条件和错误路径都有处理，不仅是成功路径

### 质量（Quality）

- [ ] 代码通过命名和结构清晰表达意图；无需注释就能理解
- [ ] 无重复业务逻辑
- [ ] 无死代码、调试输出或被注释掉的代码块
- [ ] 变更范围受限于任务；无无关的重构混在一起
- [ ] 代码通过 linting 和格式检查

### 集成（Integration）

- [ ] 变更与系统其他部分协作正常，不仅是孤立运行
- [ ] 数据库迁移、配置变更、功能开关都考虑到了
- [ ] 向后兼容性考虑周全（任何公开接口或 API 变更）

### 文档（Documentation）

- [ ] 公开接口、API、用户行为都有文档
- [ ] 值得保存的架构决策已记录（见 ADRs）
- [ ] 文档描述的是当前状态，用无时序的语言（不是"改变历史"）

### 可交付性（Ship-readiness）

- [ ] 安全影响已审查（任何不信任的输入、认证、数据处理）
- [ ] 可观测性到位（关键路径的日志、指标、链路追踪）
- [ ] 回滚方案存在（任何有风险的变更）
- [ ] 人工审查和批准已完成

## 如何应用

### 按任务（Per Task）

- 完成任务前，检查"正确性"和"质量"部分
- 确认后再标记任务完成

### 按功能（Per Feature）

- 功能完成前，检查"集成"和"文档"部分
- 进行端到端测试前

### 按发布（Per Release）

- 发布前，完整清单是底线
- `shipping-and-launch` skill 会在此基础上添加部署专属的检查

---

## 通用准则

- **一次定义，重复使用** — 项目级完成标准一旦定义，就不要每次冲刺都重新协商。标准应该是稳定的质量底线。
- **没有例外的标准** — 压力大不是跳过测试或文档的理由。标准的价值恰恰在于在时间紧张时保护质量。
- **定期审视** — 每个季度检查一下完成标准是否还适用、是否需要调整。但不要频繁改动。

## 常见的"完成"陷阱

| 陷阱 | 现实 |
|---|---|
| "已运行但未验证" | 运行不等于正确。需要测试证明。 |
| "测试通过" = 完成 | 测试通过只是完成标准的一部分。还要检查回归、文档、集成。 |
| 时间紧就降低标准 | 降低标准会导致技术债和长期延迟。保持标准。 |
| 验收条件 = 完成 | 验收条件只是"建对了"，完成标准还要"能交付"。 |
| 完成前没人工审查 | 自动化检查很重要，但人工审查也是底线。 |
